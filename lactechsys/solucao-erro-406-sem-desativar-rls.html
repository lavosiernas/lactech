Solução para o Erro 406 (Not Acceptable) sem Desativar RLS

Solução Segura: Esta solução mantém o RLS ativo enquanto corrige o erro 406 que ocorre ao tentar alterar perfis de contas secundárias.

Problema Identificado

O erro 406 (Not Acceptable) está ocorrendo quando o sistema tenta verificar a existência de usuários com o mesmo email e farm_id, excluindo o ID do usuário atual. Este tipo de consulta é necessário para o funcionamento das contas secundárias, mas está sendo bloqueado pelas políticas RLS atuais.

Exemplo de erro:

GET https://lzqbzztoawjnqwgffhjv.supabase.co/rest/v1/users?select=*&email=eq.devnasc%40gmail.com&farm_id=eq.6186a476-16a8-48c4-a158-9dd7a82f4d29&id=neq.3ee064c8-beb7-4750-931f-ea30289b5a88 406 (Not Acceptable)

Solução

A solução para este problema é modificar a política RLS da tabela users para permitir consultas específicas sem desabilitar completamente o RLS. Isso mantém a segurança do sistema enquanto resolve o problema específico das contas secundárias.

Passos para Implementação:

  1. Acesse o Dashboard do Supabase (https://app.supabase.io)
  2. Selecione o projeto: lzqbzztoawjnqwgffhjv
  3. No menu lateral, clique em "SQL Editor"
  4. Clique em "New query"
  5. Cole o conteúdo do arquivo fix_rls_users_policy.sql e execute
  6. Teste o sistema novamente para verificar se o erro 406 foi resolvido

Vantagem: Esta solução mantém o RLS ativo, preservando a segurança do sistema, enquanto resolve o problema específico das contas secundárias.

Explicação Técnica

O erro 406 (Not Acceptable) ocorre quando o Supabase não consegue retornar os dados no formato solicitado. No contexto do RLS, isso geralmente acontece quando:

  • As políticas RLS estão bloqueando o acesso aos dados
  • A consulta espera um único resultado (.single()), mas as políticas RLS estão filtrando todos os resultados
  • Há problemas de recursão nas políticas RLS

A nova política RLS criada pelo script fix_rls_users_policy.sql resolve este problema ao:

  • Permitir acesso ao próprio perfil do usuário
  • Permitir acesso a usuários da mesma fazenda
  • Adicionar uma cláusula específica para permitir consultas por email e farm_id quando o usuário pertence à mesma fazenda
